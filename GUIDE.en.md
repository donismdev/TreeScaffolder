# TreeScaffolder Technical Guide (Prompt Helper Edition)

TreeScaffolder is a **Prompt Helper and File System Scaffolding tool** designed to optimize collaboration with LLMs (Large Language Models). Its core mission is to visualize complex file structures and source code, then safely reflect them onto the actual file system.

## ⚠️ Important Warnings

1.  **No Real-time File Detection**: This program does not monitor file system changes in real-time. Do not delete, move, or modify files in the target folder via external tools (like File Explorer) while the program is open. Doing so may cause unexpected errors or data loss. (Always re-run 'Check Folder' or 'Compute Diff' after manual changes.)
2.  **Encoding Support**: All file read/write operations and recovery logs are designed based on **UTF-8** encoding. Overwriting files written in other encodings (e.g., EUC-KR, CP949) may result in corrupted characters during recovery.
3.  **Operating System**: The current version is optimized specifically for **Windows**.

---

## 1. Core Objective

TreeScaffolder is not a complex automation tool, but a **simple and safe utility designed to reduce repetitive file creation tasks**. (Optimized for Windows environments.)

-   **Easy LLM Output Application**: When applying numerous files and source codes generated by LLMs, avoid the tedious process of creating and saving each file manually by reflecting them all at once.
-   **Empty Structure Creation (Placeholders)**: Quickly generate empty folder and shell file hierarchies for initial project designs or specific requirements using only a simple text tree.
-   **Error Prevention (Visual Check)**: Focus on visually confirming and comparing which files will be modified or created before any actual disk operations occur.

---

## 2. Workflow

The recommended order of operations is as follows:

### 2.1. Source-First Analysis
Typically, you start by pasting the LLM-generated output into the `Source Code` tab.
-   **Behavior**: Analyzes `@@@FILE_BEGIN` markers within the source code to automatically extract all required parent folder structures.
-   **Benefit**: You can immediately configure a project using only the source code, without having to manually draw a tree.

### 2.2. Scaffold Tree Supplement (Tree-Secondary)
If you need specific empty folder structures or files without content yet, use the `Scaffold Tree` tab.
-   **Behavior**: Reads indentation-based tree text to plan empty files and folders.
-   **Merging**: If paths overlap between Source Code and Tree, the content from the Source Code takes precedence.

### 2.3. Compute & Name
1.  **Compute Diff (`d`)**: Integrate all input information and compare it with the actual disk.
2.  **Enter Job Name (`4`)**: Provide a descriptive name for this task (e.g., `Feature_Login_System`). This name is permanently recorded in the log files.

### 2.4. Review & Apply
1.  **Check After View**: Verify status colors: Green (New), Blue (Change), Red (Potential Data Loss).
2.  **Filter with Checkboxes**: Deselect any items you wish to exclude from the operation.
3.  **Apply Scaffold (`f`)**: Reflect the plan onto the actual file system.

### 2.5. File-Based Template Loading
For frequently used structures or code, you can prepare them in text files and load them instantly.
-   **Method**: Write your content in `Resources/sample_blueprint.txt` (for Source Code) and `Resources/sample_structure.txt` (for Tree Structure).
-   **Execution**: Click the **Load Test Data** button at the bottom to immediately populate the editors with the file contents.
-   **Benefit**: This is a very convenient way to manage predefined templates without the need for manual copy-pasting every time.

## 3. V2 Multipatch Format (@@@ Blocks)

TreeScaffolder uses a unique block format for seamless communication with LLMs.

### Block Types
-   **FILE**: Defines the path and content of an actual file.
-   **PATCH**: Modifies specific parts of a file. (Supports FIND, REPLACE, INSERT_AFTER, INSERT_BEFORE, etc.)
-   **COMMENT**: Contains documentation info like logs, descriptions, or the Unified Scaffold Structure.

### Rules
-   Every block starts with `@@@KEYWORD_BEGIN <Path>` and ends with `@@@KEYWORD_END`.
-   Content is extracted **verbatim** without modification and saved to files.

## 7. Indentation Principle

TreeScaffolder interprets tree structures based on **Strict Python-style Indentation**.

-   **Consistency**: Within a single tree, you must use ONLY **Tabs (\t)** or ONLY a **consistent number of Spaces**. Mixing both will result in a `TabError`.
-   **Unit Detection**: The tool automatically detects the indentation unit (number of spaces or tabs) from the first indented level (Depth 1) and enforces it for the rest of the tree.
-   **Recommendation**: When collaborating with LLMs, using **Tabs (\t)** is highly recommended to avoid invisible whitespace issues.
