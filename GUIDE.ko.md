# 🌲 Tree Scaffolder 기술 설명서

이 문서는 "Tree Scaffolder" 도구의 내부 동작, 데이터 흐름, 핵심 로직을 상세히 설명하여 향후 유지보수 및 기능 추가 시 참조할 수 있도록 작성되었습니다.

---

## 1. 핵심 목적

텍스트로 정의된 트리 구조와 파일 내용을 기반으로, 지정된 경로에 디렉토리와 파일을 생성(스캐폴딩)하는 개발자용 생산성 도구입니다.

---

## 2. 구성 요소 (핵심 스크립트)

-   `gui_app.py`
    -   **역할**: 메인 GUI 애플리케이션 (Tkinter 기반).
    -   **기능**: 사용자 입력 관리, `Plan` 요청, 스캐폴딩 실행 제어, 로그 출력.
-   `Scripts/Core/scaffold_core.py`
    -   **역할**: 비즈니스 로직의 핵심 엔진 (GUI와 독립적).
    -   **기능**: 트리 텍스트 파싱, V2 패치 파싱 요청, 파일 시스템 분석 및 차이점(Diff) 계산, `Plan` 객체 생성.
-   `Scripts/Core/v2_parser.py`
    -   **역할**: V2 멀티패치 포맷 전용 파서.
    -   **기능**: `@@@FILE_BEGIN` 등 블록 구문을 파싱하여 파일 경로와 내용 추출.
-   `Scripts/UI/scaffold_runner.py`
    -   **역할**: 실제 파일 시스템 수정 작업 수행.
    -   **기능**: 디렉토리 생성, 파일 쓰기, 작업 로그 기록 및 통계 요약.

---

## 3. 핵심 규칙 및 문법

### 3.1. 트리 구조 (Scaffold Tree)

-   **인덴트**: 탭(Tab) 또는 스페이스(Space)를 사용한 계층 구조를 정의합니다. (두 방식의 혼용은 금지됩니다.)
-   **폴더**: 이름 끝에 `/` 또는 `\`가 붙으면 폴더로 인식합니다.
-   **파일**: 폴더가 아닌 항목은 파일로 인식합니다.
-   **루트 마커**: `@ROOT {{Root}}`와 같이 루트를 지정하며, 트리의 첫 항목은 이 마커와 일치해야 합니다.

    #### 예시
    ```text
    @ROOT {{Root}}
    {{Root}}/
        NewModule/
            Public/
                NewModule.h
            Private/
                NewModule.cpp
            NewModule.Build.cs
    ```

### 3.2. 블록 구문 (V2)

-   **목적**: 파일 내용, 주석 등 특정 목적을 가진 텍스트 블록을 정의합니다.
-   **규칙**:
    -   `@@@<KEYWORD>_BEGIN [선택적 식별자]`로 블록 내용 정의를 시작합니다. `<KEYWORD>`는 `FILE`, `COMMENT` 등 예약된 키워드입니다.
    -   `@@@<KEYWORD>_END`로 내용 정의를 끝냅니다. 시작 태그와 `KEYWORD`가 일치해야 합니다.
    -   **가장 중요한 규칙**: 파서는 `@@@..._BEGIN`과 `@@@..._END`로 명확하게 감싸인 블록만 인식합니다. **블록 외부의 모든 텍스트는 주석으로 간주되어 완전히 무시됩니다.**
    -   모든 `BEGIN`과 `END` 태그는 반드시 올바른 순서로 쌍을 이뤄야 하며, 블록 안에 다른 블록이 중첩될 수 없습니다. 위반 시 오류가 발생합니다.
    -   `FILE`처럼 인식된 키워드의 블록 내용은 파서가 사용하고, `COMMENT`처럼 무시하도록 설정된 키워드의 블록 내용은 건너뜁니다. 인식되지 않은 키워드의 블록도 무시됩니다.

#### 예시: 파일 내용 블록 (`FILE`)

```
@@@FILE_BEGIN {{Root}}/NewModule/NewModule.Build.cs
using UnrealBuildTool;

public class NewModule : ModuleRules
{
    public NewModule(ReadOnlyTargetRules Target) : base(Target)
    {
        // ...
    }
}
@@@FILE_END
```

#### 예시: 주석 블록 (`COMMENT`)

```
이 텍스트는 무시됩니다.

@@@COMMENT_BEGIN 이 블록은 파서에 의해 인식되지만, 내용은 결과물에 포함되지 않습니다.
- 빌드 과정에 대한 메모
- 임시 비활성화된 코드
등을 여기에 작성할 수 있습니다.
@@@COMMENT_END

이 텍스트 또한 무시됩니다.
```

### 3.3. 줄 바꿈 처리 (Line Endings)

-   **표준**: 이 도구는 **Windows 표준(CRLF, `\r\n`)** 줄 바꿈 형식을 따릅니다.
-   **강제 적용**: `@@@FILE` 블록을 통해 생성되는 모든 파일은 내용의 원본 형식과 관계없이 **강제로 CRLF로 변환**되어 저장됩니다. 
-   **이유**: Unreal Engine (C++, C#) 및 다양한 Windows 개발 환경에서의 호환성을 보장하기 위함입니다.

### 3.4. 에디터 툴바 (Editor Toolbar)

"Define Scaffold Tree" 패널의 텍스트 에디터 상단에는 동적으로 상태가 변경되는 툴바가 존재합니다.

-   **동작 규칙**: 툴바의 버튼들은 현재 선택된 탭("Scaffold Tree", "Source Code", "Content")에 따라 활성화 또는 비활성화됩니다.
-   **Scaffold Tree / Content 탭**: 모든 툴바 버튼이 비활성화됩니다.
-   **Source Code 탭**:
    -   `make tree` 버튼이 활성화됩니다.
    -   **자동 병합 (Implicit Merge)**: 실제 스캐폴딩 실행(`generate_plan`) 시, "Scaffold Tree" 에디터의 텍스트와 "Source Code" 에디터의 V2 블록 정보는 **논리적으로 병합**됩니다. 따라서 "Source Code"에만 정의된 파일이라도 `make tree`를 거치지 않고 바로 스캐폴딩을 수행하면 정상적으로 생성됩니다.
    -   **`make tree` 기능**:
        1.  현재 "Source Code" 에디터에 있는 모든 V2 블록 (`@@@FILE_BEGIN...END` 등)을 파싱합니다.
        2.  추출된 모든 `path` 정보를 기반으로, "Scaffold Tree"에 맞는 새로운 트리 구조 텍스트를 자동으로 생성합니다.
        3.  기존 "Scaffold Tree"의 내용을 모두 지우고, 새로 생성된 트리 텍스트로 덮어씁니다.
        4.  작업 완료 후 "Scaffold Tree" 탭으로 자동 전환하여 결과를 바로 확인할 수 있습니다.
        5.  이 기능은 V2 블록이 이미 작성된 텍스트로부터 트리 구조를 역으로 생성할 때 유용합니다.

## 4. 데이터 흐름 및 실행 순서 (GUI 기준)

1.  **사용자 입력**: 사용자가 GUI의 "Scaffold Tree" 또는 "Source Code" 텍스트 영역에 내용을 입력하고 "Compute Diff"를 클릭합니다.
2.  **계획 생성 (`generate_plan`)**:
    -   `gui_app.py`는 두 텍스트 영역의 내용을 합쳐 `scaffold_core.generate_plan` 함수에 전달합니다.
    -   `scaffold_core`는 먼저 트리 구문을 파싱하여 기본적인 디렉토리/파일 구조(`planned_dirs`, `planned_files`)를 계획합니다.
    -   그 다음 `v2_parser.parse_v2_format`를 호출하여 V2 구문에서 파일 경로와 내용을 추출하고 `file_contents` 딕셔너리에 저장합니다.
    -   파일 시스템을 스캔하여 계획된 경로의 현재 상태(`new`, `exists`, `overwrite`, `conflict`)를 `path_states` 딕셔너리에 기록합니다.
    -   이 모든 정보가 담긴 `Plan` 객체가 `gui_app.py`로 반환됩니다.
3.  **결과 시각화**: `gui_app.py`는 `Plan` 객체를 사용하여 "Before"와 "After" 트리 뷰를 그립니다. `new`, `overwrite` 등의 상태에 따라 다른 색상과 아이콘이 표시됩니다.
    -   **트리 선택 하이라이트**: 사용자가 트리 항목을 클릭하면 해당 트리 뷰가 포커스를 얻으며 **진한 파란색**으로 강조됩니다. 이때 다른 쪽 트리 뷰의 선택 항목은 **회색**으로 변하여, 현재 `Content` 탭에 표시되는 내용이 어느 쪽 트리에서 온 것인지 시각적으로 구분해 줍니다.
4.  **스캐폴드 적용 (`on_apply`)**:
    -   사용자가 "Apply Scaffold"를 클릭합니다.
    -   **확인 메시지 및 요약**:
        -   **Dry Run 모드**: 실제 파일이 생성되지 않음을 안내하고, 시뮬레이션될 폴더/파일/덮어쓰기 개수를 요약하여 보여줍니다.
        -   **일반 모드**: 실제 파일 시스템 변경에 대한 **강력한 경고(⚠️)**와 함께, 생성 및 덮어쓰기될 항목의 정확한 개수를 사용자에게 확인받습니다.
    -   사용자가 동의하면 `_execute_scaffold` 메소드가 호출됩니다.
5.  **파일 시스템 쓰기 (`_execute_scaffold`)**:
    -   계획된 디렉토리를 먼저 생성합니다 (`_ensure_dir`).
    -   그 다음, 계획된 파일을 생성/덮어쓰기 합니다 (`_ensure_file`).
    -   `_ensure_file` 함수는 `path.write_bytes()`를 사용하여 파일을 씁니다. 이때 **강제로 CRLF(`\r\n`) 변환**이 수행됩니다. 이 방식은 **UTF-8 (BOM 없음)** 인코딩을 보장하며, 기존 파일이 있다면 내용을 완전히 덮어씁니다.
6.  **완료**: 모든 작업이 끝나면 로그가 기록되고, "Before/After" 뷰가 새로고침됩니다.

## 5. 중요 로직 상세

### 파일 덮어쓰기 (Overwrite) 로직

-   **판단 기준**: `scaffold_core.generate_plan`에서 파일 시스템의 경로(`path`)가 이미 존재하고(`path.exists()`), 계획된 파일이며(`path in plan.planned_files`), V2 포맷으로 파일 내용까지 정의되었을 때(`path.resolve() in plan.file_contents`), 해당 파일의 상태는 `'overwrite'`가 됩니다.
-   **실행**: `_ensure_file` 함수는 `is_overwrite` 플래그가 True일 때, 파일을 열고 새로운 내용으로 덮어씁니다. 이때 줄 바꿈은 CRLF로 통일됩니다.

### 블록 내용 처리 (Verbatim)

-   **규칙**: `v2_parser.py`는 `@@@<KEYWORD>_BEGIN`과 `@@@<KEYWORD>_END` 태그 사이의 내용을 어떤 수정(예: 공백 다듬기) 없이 **있는 그대로(verbatim)** 추출합니다. 단, 파일 저장 시점에 줄 바꿈은 시스템 표준(CRLF)으로 정규화됩니다.

## 6. 환경 설정 (`config.json`)

애플리케이션의 동작을 제어하는 설정 파일입니다.

-   `debug_level` (integer, 0 ~ 3)
    -   로그의 상세 정도를 제어합니다.
    -   **`0` (Release)**: 치명적인 오류와 경고만 기록합니다.
    -   **`1` (Info)**: 기본값. 주요 실행 단계와 결과를 기록합니다.
    -   **`2` (Debug)**: 개발 및 문제 해결을 위한 상세 정보를 기록합니다.
    -   **`3` (Trace)**: 아주 사소한 동작까지 포함한 모든 내용을 기록합니다.
-   **런타임 로그 (자동 기록)**
    -   애플리케이션 실행 시, 모든 동작과 오류를 기록하는 런타임 로그가 항상 생성됩니다.
    -   각 실행 세션은 `Log/Session_YYYYMMDD_HHMMSS/` 폴더에 독립적으로 저장됩니다.
    -   애플리케이션 종료 시, 로그 파일의 최상단에 스캐폴드 적용 여부(실행됨/실행되지 않음)가 자동으로 추가됩니다.
-   `log_cleanup_limit` (integer): 옵션 UI에서 로그 정리 시 보관할 최신 로그 세션 폴더의 개수입니다. 기본값은 5입니다.

## 8. 인터랙티브 기능 및 분석 (Analysis Details)

### 8.1. After View 개별 항목 선택 (Checkboxes)
- **목적**: 계획된 전체 항목 중 사용자가 원하는 항목만 선택하여 점진적으로 적용할 수 있습니다.
- **표시**: 신규 파일 생성, 덮어쓰기, 충돌 항목 앞에 체크박스(`☑`/`☐`)가 표시됩니다.
- **조작**:
    - **재클릭 토글**: 이미 선택된 항목을 한 번 더 클릭하면 체크박스가 토글됩니다. (첫 클릭은 내용 확인 및 동기화 용도)
    - **Space바**: After View에 포커스가 있을 때 Space바를 눌러 선택 상태를 토글할 수 있습니다.
- **자동 연동 (Recursive Logic)**:
    - **부모 선택**: 하위 항목을 체크하면, 해당 경로를 생성하기 위해 필요한 모든 상위 부모 폴더가 자동으로 함께 선택됩니다.
    - **부모 해제**: 상위 폴더를 체크 해제하면 그 안의 모든 하위 항목(폴더 및 파일)이 생성 단계에서 자동으로 건너뛰어집니다.
- **요약 반영**: 체크 상태를 변경하면 하단 `Summary` 패널의 숫자(폴더/파일 개수)가 실시간으로 업데이트됩니다.

### 8.2. Before/After 뷰 동기화 및 컨텐츠 출처
- **기능**: After View에서 특정 항목을 클릭하면, Before View에서도 해당 항목을 자동으로 찾아 선택해 줍니다.
- **엄격한 컨텐츠 출처**:
    - **Before View**: 언제나 **실제 디스크에 저장된 현재 파일 내용**만 보여줍니다.
    - **After View**: 언제나 **에디터에 입력된 계획된 내용**만 보여줍니다. 디스크 내용을 참조하지 않으므로, '현재 상태'와 '복구/계획된 상태'를 명확하게 비교할 수 있습니다.
- **제한**: 더블클릭 동작은 클릭 조작의 안정성을 위해 비활성화되어 있습니다. 트리를 조작하려면 왼쪽의 화살표 아이콘을 사용하십시오.

### 8.3. 안전 기능 (자동 백업)
- **자동 복구 로그 생성**: 최후의 방어선으로, 적용 단계에서(Dry Run 제외) 기존 파일을 덮어쓰기 직전에 모든 대상 파일의 원본 내용을 `Log` 폴더 내의 V2 스타일 로그 파일 하나로 모아서 저장합니다.
- **목적**: 실수로 중요한 파일이 포함된 스캐폴딩을 적용했더라도, 'Recovery' 도구를 사용하여 이전 상태로 간편하게 복구할 수 있도록 보장합니다.

### 8.4. 복구 도구 (Recovery Tool)
- **개요**: 실행 단계에서 자동으로 생성된 백업 로그(`scaffold_recovery_*.txt`)를 사용하여 파일이나 소스 코드를 이전 상태로 되돌리는 내장 유틸리티입니다.
- **작동 방식**:
    1. 하단 요약 패널의 **"Recovery"** 버튼을 클릭하여 복구 창을 엽니다.
    2. 도구가 `Log` 폴더에서 사용 가능한 `scaffold_recovery_*.txt` 형식의 파일을 검색하여 목록으로 보여줍니다.
    3. 목록에서 로그 파일을 선택하면 덮어쓰기 전의 파일 상태를 확인할 수 있습니다.
    4. **선택적 복구**: After View의 체크박스를 사용하여 복구하고 싶은 특정 파일이나 폴더를 선택할 수 있습니다.
    5. **"Apply Recovery"** 버튼을 눌러 선택한 항목을 원래 상태로 되돌립니다.
- **소스 코드 복구**: 복구 로그를 로드하면 메인 창의 "Source Code" 에디터도 해당 로그의 V2 포맷 내용으로 채워집니다. 이를 통해 필요한 경우 이전 코드를 수동으로 복사하거나 수정할 수 있습니다.
- **안전성**: 복구 도구 역시 메인 애플리케이션과 동일하게 "미리보기 -> 최종 확인" 워크플로우를 따르므로, 복구 과정에서 발생할 수 있는 데이터 손실을 방지합니다.

## 9. 단축키 바인딩 규칙 및 제한사항

`Resources/key_bindings_map.json` 파일에 단축키를 정의할 때 다음 규칙 및 제한사항을 참고하십시오.

-   **일반 키**: 대부분의 알파벳(`a`~`z`), 숫자(`0`~`9`), 특수 문자(`-`, `=`, `,` 등)는 단일 키로 바인딩할 수 있습니다.
-   **특수 키**:
    -   `Space`, `Escape`, `Grave` (백틱, `` ` ``)과 같은 특수 키는 위에 명시된 형태로 바인딩할 수 있습니다.
    -   `F1`부터 `F12`까지의 기능 키 (`F5` 등)도 바인딩 가능합니다.
-   **조합 키 (Modifier Keys)**:
    -   `Control` (`Ctrl`), `Alt`, `Shift` 키는 단독으로 기능을 수행하기보다는 다른 키와 조합하여 사용됩니다 (예: `Control+S`, `Alt+F`, `Shift+Escape`).
    -   `key_bindings_map.json`에서 조합 키를 사용할 때는 `+`로 연결합니다 (예: `"Control+S"`).
    -   **`Alt` 키의 특수성**:
        -   `Alt` 키는 운영체제 수준에서 메뉴 활성화 등 다양한 시스템 기능에 예약되어 있는 경우가 많습니다.
        -   이 애플리케이션에서는 `Alt` 키를 단독으로 누르면 현재 화면에 활성화된 단축키 힌트가 표시되는 기능에 사용됩니다. 따라서 `Alt` 키 단독 또는 `Alt` 키와 조합된 일부 단축키는 예상대로 동작하지 않거나, 시스템 기능과 충돌할 수 있습니다.
        -   예를 들어, `"Alt"` 또는 `"Alt+C"`와 같이 `Alt` 키가 포함된 단축키를 지정하더라도, Tkinter 또는 운영체제가 해당 이벤트를 가로챌 수 있습니다. 따라서 `Alt` 키를 메인 액션 단축키로 사용하는 것은 권장되지 않습니다.

-   **불가능한 바인딩**:
    -   마우스 클릭 이벤트 (`<Button-1>`, `<Double-Button-1>` 등)는 현재 `key_bindings_map.json`을 통한 단축키로 직접 매핑하기 어렵습니다.
    -   Tkinter 및 운영체제의 제약으로 인해 일부 키 조합은 바인딩이 불가능하거나 일관되지 않은 동작을 보일 수 있습니다.

이 문서는 `key_bindings_map.json` 파일에 사용되는 단축키 표기를 위한 가이드입니다.

## 10. 설계 철학 및 유지보수 원칙

### 10.1. 안전 우선 (Safety Over Aesthetics)
이 애플리케이션은 사용자의 파일을 직접 삭제하거나 덮어쓰는 매우 위험한 작업을 수행합니다. 따라서 코드가 조금 '못생겼거나' '비효율적'으로 보이더라도, 수많은 시행착오를 거쳐 검증된 **경험적(Empirical) 로직**을 최우선으로 존중합니다.

- **AI 리팩토링 금지**: AI를 이용한 기계적인 코드 정리나 최적화는 100% 확률로 예외 상황(Edge Case)을 깨뜨리는 결과를 초래했습니다. 기존 로직에 대한 자의적인 '개선' 시도는 절대 금지됩니다.
- **성능보다 정확성**: 초 단위의 성능 최적화는 이 프로젝트에서 무의미합니다. 시스템 부하가 크지 않은 작업이므로, 조금 느리더라도 가장 확실하고 안전한 방법을 택합니다.
