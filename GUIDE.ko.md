# TreeScaffolder 기술 설명서 (Prompt Helper Edition)

TreeScaffolder는 LLM(대규모 언어 모델)과의 협업을 최적화하기 위해 설계된 **프롬프트 헬퍼 및 파일 시스템 스캐폴딩 도구**입니다. 이 도구는 복잡한 파일 구조와 소스 코드를 시각화하고, 이를 실제 파일 시스템에 안전하게 반영하는 것을 핵심 목적으로 합니다.

## ⚠️ 중요 주의사항 (Important Warnings)

1.  **실시간 파일 감지 불가**: 본 프로그램은 파일 시스템의 변경 사항을 실시간으로 감시하지 않습니다. 프로그램을 열어둔 상태에서 외부(탐색기 등)를 통해 대상 폴더의 파일을 삭제, 이동, 혹은 수정하지 마십시오. 외부 조작 후 작업을 계속할 경우 예기치 않은 오류가 발생하거나 데이터가 손실될 수 있습니다. (변경 시 반드시 '폴더 체크' 혹은 '계획 계산'을 다시 수행하십시오.)
2.  **인코딩 지원**: 모든 파일 읽기/쓰기 및 복구 로그 생성은 **UTF-8** 인코딩을 기준으로 설계되었습니다. EUC-KR, CP949 등 다른 인코딩으로 작성된 파일을 덮어쓸 경우, 복구 과정에서 인코딩 깨짐 현상이 발생할 수 있습니다.
3.  **운영체제**: 현재 버전은 **Windows** 환경에 최적화되어 설계되었습니다.

## 1. 핵심 목적 (Core Objective)

TreeScaffolder는 거창한 자동화 도구가 아닌, **반복되는 파일 생성 작업을 줄여주는 단순하고 안전한 보조 유틸리티**입니다. (Windows 환경에 최적화되어 있습니다.)

-   **LLM 결과물 적용 편의성**: LLM(ChatGPT, Claude 등)이 생성해준 수많은 파일과 소스 코드를 프로젝트에 반영할 때, 직접 파일을 하나씩 만들고 저장하는 번거로운 과정을 클릭 한 번으로 해결합니다.
-   **빈 구조 생성 (Placeholder)**: 프로젝트 초기 설계나 특정 폴더 구조가 필요할 때, 텍스트 트리만 작성하여 빈 폴더와 껍데기 파일들을 즉시 생성할 수 있습니다.
-   **실수 방지 (Visual Check)**: 실제 파일을 디스크에 쓰기 전, 어떤 파일이 바뀌거나 새로 생기는지 시각적으로 대조하고 확인하는 과정에 집중합니다.

## 2. 사용 방법 (Workflow)

가장 권장되는 사용 순서는 다음과 같습니다.

### 2.1. 소스 코드 기반 분석 (Source-First)
많은 경우 소스 코드 에디터(`Source Code` 탭)에 LLM이 생성한 결과물을 붙여넣는 것으로 작업을 시작합니다.
-   **동작**: 소스 코드 내의 `@@@FILE_BEGIN` 마커를 분석하여 필요한 모든 부모 폴더 구조를 자동으로 추출합니다.
-   **장점**: 별도로 트리를 그릴 필요 없이 소스 코드만으로 즉시 프로젝트를 구성할 수 있습니다.

### 2.2. 스캐폴드 트리 보완 (Tree-Secondary)
특정 빈 폴더 구조나 아직 내용이 없는 파일이 필요하다면 `Scaffold Tree` 탭을 사용합니다.
-   **동작**: 들여쓰기 기반의 트리 텍스트를 읽어 빈 파일과 폴더 계획을 세웁니다.
-   **병합**: 소스 코드와 트리에 중복된 경로가 있다면 소스 코드의 내용이 우선 적용됩니다.

### 2.3. 계획 계산 및 이름 지정 (Compute & Name)
1.  **Compute Diff (`d`)**: 입력된 모든 정보를 통합하여 실제 디스크와 비교합니다.
2.  **작업 이름 입력 (`4`)**: 이번 작업의 성격을 나타내는 이름을 입력합니다 (예: `Feature_Login_System`). 이 이름은 로그 파일에 영구 기록됩니다.

### 2.4. 최종 검토 및 적용 (Apply)
1.  **After View 확인**: 녹색(신규), 파란색(변경), 빨간색(유실 주의) 색상을 확인합니다.
2.  **체크박스 필터링**: 원하지 않는 항목은 체크 해제하여 제외할 수 있습니다.
3.  **Apply Scaffold (`f`)**: 실제 파일 시스템에 반영합니다.

### 2.5. 파일 기반 템플릿 활용 (File-Based Loading)
자주 사용하는 구조나 코드가 있다면 파일에 미리 작성해두고 빠르게 불러올 수 있습니다.
-   **방법**: `Resources/sample_blueprint.txt` (소스 코드용) 및 `Resources/sample_structure.txt` (트리 구조용) 파일에 내용을 작성합니다.
-   **실행**: 하단의 **Load Test Data** 버튼을 누르면 해당 파일의 내용이 에디터에 즉시 로딩됩니다. 
-   **장점**: 매번 복사/붙여넣기를 할 필요 없이, 템플릿 파일을 관리하는 것만으로도 매우 편리하게 작업할 수 있습니다.

## 3. V2 멀티패치 포맷 (@@@ Blocks)

LLM과의 원활한 소통을 위해 TreeScaffolder는 고유의 블록 포맷을 사용합니다.

### 블록 종류
-   **FILE**: 실제 파일의 경로와 내용을 정의합니다.
-   **COMMENT**: 로그, 설명, 또는 Unified Scaffold Structure와 같은 문서화 정보를 담습니다.

### 규칙
-   모든 블록은 `@@@KEYWORD_BEGIN <Path>`로 시작하고 `@@@KEYWORD_END`로 끝납니다.
-   내용은 수정 없이 **있는 그대로(verbatim)** 추출되어 파일로 저장됩니다.

## 4. LLM 협업용 프롬프트 가이드 (Prompting Guide)

LLM으로부터 TreeScaffolder에 바로 붙여넣을 수 있는 최적의 결과물을 얻으려면 아래 프롬프트를 복사하여 LLM에게 전달하십시오.

### **추천 프롬프트 (Copy & Paste)**

모든 산출물은 'Multi-File Patch Format v2'로만 제공해야 하며, 아래 규칙을 엄격히 따른다.

[출력 형식]
- 응답은 반드시 단 하나의 코드블록( ```text )으로 작성한다.
- 블록 안에는 오직 v2 패치 내용만 포함한다. (설명/목차/제목 금지)

[v2 패치 규칙 (중요: {{Root}} 마커 필수)]
1) 각 파일은 아래 형태로 출력한다. 경로 시작은 반드시 {{Root}} 마커를 사용한다.

- [여기에 사용자의 특정 코딩 스타일이나 언어 표준을 추가하십시오]

[트리 구조 예시]

## 5. 환경 설정 (`config.json`)

-   `debug_level` (0~3): 2 이상 설정 시 **[폴더 체크]** 디버그 버튼이 활성화됩니다.
-   `log_cleanup_limit`: 보관할 최근 로그 세션 폴더의 개수입니다.

## 6. 키보드 단축키 (Quick Reference)

| 키 | 동작 | 설명 |
| :--- | :--- | :--- |
| `d` | **Compute Diff** | 현재 입력을 기반으로 스캐폴딩 계획 계산 |
| `4` | **Focus Job Name** | 작업 이름 입력창으로 이동 (분석 후 활성화) |
| `f` | **Apply Scaffold** | 계획을 파일 시스템에 적용 (확인 창 출력) |
| `q` | **Recovery** | 자동 백업된 복구 로그 창 열기 |
| `r` | **Clear Data** | 모든 입력 내용 및 계획 초기화 |
| `e` | **Prev Folder** | 이전 작업 폴더 경로 불러오기 |
| `w` | **Browse Folder** | 새 작업 폴더 선택 |
| `o` | **Options** | 언어 및 디버그 설정 창 열기 |
| `Space` | **Toggle** | (After View) 선택 항목 체크박스 토글 |

## 7. 들여쓰기 원칙 (Indentation Principle)

TreeScaffolder는 **파이썬 스타일의 엄격한 들여쓰기**를 기반으로 트리 구조를 해석합니다.

-   **일관성**: 한 트리 내에서는 오직 **탭(\t)**만 쓰거나, 오직 **N칸의 공백(Space)**만 사용해야 합니다. 둘을 섞어 쓰면 `TabError`가 발생합니다.
-   **단위 인식**: 첫 번째 들여쓰기 단계(Depth 1)에서 사용된 공백의 개수나 탭의 개수를 자동으로 감지하여 해당 프로젝트의 '기본 들여쓰기 단위'로 고정합니다.
-   **에디터 권장**: LLM과 협업 시에는 보이지 않는 공백 문제를 방지하기 위해 **탭(\t)** 사용을 강력히 권장합니다.

## 8. 설계 철학: 안전 우선 (Safety First)

이 도구는 **경험적(Empirical) 로직**을 존중합니다.
-   **AI 리팩토링 절대 금지**: 기계적인 최적화보다 수많은 시행착오로 검증된 안전 코드를 우선시합니다.
-   **성능보다 정확성**: 0.1초 빠른 처리보다 내 파일이 안전하게 보호되는 것이 더 중요합니다.
