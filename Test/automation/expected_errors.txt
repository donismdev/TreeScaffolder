# TreeScaffolder v0.8+ 코드 진단 및 예상 오류 (Diagnostic Report)

본 문서는 현재 코드의 구조적 결함으로 인해 테스트가 어렵거나 사이드 이팩트가 발생할 수 있는 지점을 정리합니다.

## 1. 테스트 불가능한 구조 (Non-Testable Patterns)

### 1.1. 높은 UI 결합도 (UI Tight Coupling)
- **문제**: `execute_scaffold`, `on_recompute`, `on_check_folder` 등의 핵심 로직이 `app` (Tkinter 객체)을 직접 인자로 받아 UI 위젯의 값을 읽거나(`get()`) 직접 수정합니다.
- **영향**: Tkinter 루프를 돌리지 않고는 로직만 따로 떼어내어 단위 테스트(Unit Test)를 수행하기가 매우 어렵습니다.
- **개선 필요**: UI 위젯 대신 순수 데이터 객체(Data Class 등)를 인자로 받도록 분리해야 합니다.

### 1.2. 내부 함수(Nested Functions) 및 캡슐화 위반
- **문제**: `execute_scaffold` 내부에 `log_exec` 함수가 정의되어 있어, 외부 테스트 스크립트에서 이 로그가 정상적으로 쌓이는지 가로채서 확인할 방법이 없습니다.
- **영향**: 실행 과정의 정밀 검증이 불가능합니다.

### 1.3. 부수 효과(Side Effects)의 직접 실행
- **문제**: 로직 중간에 `messagebox.showerror`가 직접 호출됩니다.
- **영향**: 자동화 테스트 도중 팝업이 뜨면서 테스트가 중단(Hang)되거나 실패합니다.

## 2. 잠재적 논리 에러 및 위험 (Potential Bugs)

### 2.1. 윈도우 경로 대소문자 무시 위험 (Windows Case-Insensitivity)
- **현상**: `path_states` 딕셔너리의 키로 `Path` 객체를 사용 중이나, `resolve()`된 결과가 시스템 설정이나 타이밍에 따라 대소문자가 달라질 수 있습니다.
- **위험**: 동일 파일에 대해 'New'와 'Exists' 상태가 동시에 존재하는 '고스트 노드' 현상이 발생할 수 있습니다.

### 2.2. 복구 로그(Recovery)의 무결성 보장 미흡
- **현상**: 복구 로그 작성 시 `errors='replace'`를 사용하여 인코딩 오류를 회피합니다.
- **위험**: 사용자는 복구가 성공했다고 생각하지만, 실제로는 깨진 글자(`?`)가 포함된 데이터가 저장될 수 있습니다.

### 2.3. .gitkeep 생성 시점과 통계 불일치
- **현상**: `.gitkeep`은 `execute_scaffold` 도중 실시간으로 생성되지만, 상단 요약(Summary)은 실행 시작 시점의 계획을 기반으로 합니다.
- **위험**: 실제 생성된 개수와 사용자가 화면에서 본 계획 개수가 다르게 표기될 수 있습니다. (현재 0.8 버전에서 일부 보완되었으나 시점 차이 존재)

### 2.4. 트리 인덴트 점프 허용 (Indentation Jump) - [CRITICAL]
- **현상**: 부모 노드보다 2단계 이상 깊은 인덴트가 바로 등장해도 에러 없이 부모 바로 아래에 생성함. (예: Depth 0 -> Depth 3)
- **위험**: 사용자의 오타나 구조적 실수가 교정되지 않고 잘못된 위치에 파일이 생성됨.

## 3. 테스트 친화적 코드 개선 제안 (Strategy)
1. **Dependency Injection**: 로그 출력 함수(`log_exec`)를 인자로 전달받도록 변경.
2. **Logic Extraction**: 파일 생성 로직과 UI 업데이트 로직을 엄격히 분리.
3. **Mocking-ready**: `messagebox` 호출부를 별도의 UI 유틸리티로 래핑(Wrapping)하여 테스트 시 가짜(Mock) 함수로 치환 가능하게 함.
